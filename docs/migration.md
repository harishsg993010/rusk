# Migration Guide

rusk works with existing projects out of the box. In most cases, you don't need to "migrate" at all -- just run `rusk install` in your project directory.

---

## From npm

### Option A: Just use rusk install

If your project has a `package.json`, rusk reads it directly:

```bash
cd my-npm-project/    # has package.json
rusk install          # resolves from package.json, creates rusk.lock
```

rusk creates `rusk.lock` alongside `package-lock.json`. You can keep both lockfiles during the transition, or delete `package-lock.json` when you're ready.

### Option B: rusk migrate

Import dependencies from `package-lock.json` into a `rusk.toml`:

```bash
cd my-npm-project/
rusk migrate
# Migrated 70 packages from npm (package-lock.json)
# Created rusk.toml
# Run 'rusk install' to resolve, download, and create rusk.lock.

rusk install
```

`rusk migrate` pins dependencies to the exact versions from `package-lock.json`. After migration, you have a `rusk.toml` with all dependencies listed.

---

## From yarn

```bash
cd my-yarn-project/
rusk migrate --from yarn
# Migrated 85 packages from yarn (yarn.lock)
# Created rusk.toml

rusk install
```

If you omit `--from`, rusk auto-detects by looking for `yarn.lock`.

---

## From pnpm

```bash
cd my-pnpm-project/
rusk migrate --from pnpm
# Migrated 62 packages from pnpm (pnpm-lock.yaml)
# Created rusk.toml

rusk install
```

If your project uses pnpm's isolated node_modules layout, you can replicate it with:

```toml
[js_dependencies]
node_linker = "isolated"
```

---

## From pip

No migration command needed. rusk reads `requirements.txt` directly:

```bash
cd my-pip-project/    # has requirements.txt
rusk install          # reads requirements.txt, creates rusk.lock
```

After install, packages land in `.venv/lib/site-packages/`. If you don't have a `.venv` yet:

```bash
rusk venv             # creates .venv
rusk install          # installs into .venv
```

`rusk add` and `rusk remove` read and write `requirements.txt` when no other manifest is present.

---

## From poetry

rusk reads `pyproject.toml` with PEP 621 `[project.dependencies]` and Poetry-style `[tool.poetry.dependencies]`:

```bash
cd my-poetry-project/    # has pyproject.toml
rusk install             # reads pyproject.toml, creates rusk.lock
```

If your `pyproject.toml` uses Poetry format:

```toml
[tool.poetry.dependencies]
python = "^3.9"
flask = "^3.0.0"
requests = "^2.28.0"
```

rusk understands this and resolves accordingly. You can keep using `pyproject.toml` as your manifest -- no `rusk.toml` required.

---

## From uv

rusk reads the same `pyproject.toml` format that uv uses:

```bash
cd my-uv-project/    # has pyproject.toml
rusk install         # reads pyproject.toml, creates rusk.lock
```

The main difference: rusk creates `rusk.lock` instead of `uv.lock`. The lockfile format is different (rusk includes SHA-256 digests, signature metadata, and provenance information for every package).

If your project uses `requirements.txt` generated by `uv pip compile`, that works too:

```bash
cd my-uv-project/
rusk install         # reads requirements.txt
```

---

## What happens to existing lockfiles?

rusk never modifies foreign lockfiles. It creates its own `rusk.lock` alongside whatever already exists:

```
my-project/
├── package.json           # kept as-is
├── package-lock.json      # kept as-is (npm)
├── rusk.lock              # new (rusk)
└── node_modules/          # managed by rusk
```

You can delete the old lockfile when you're confident in the switch. Or keep both -- rusk ignores foreign lockfiles entirely.

---

## Differences to expect

**Signatures and provenance.** rusk verifies signatures and provenance by default (when available) and reports the results. Other tools don't. Run `rusk audit` to see the trust state of your dependencies.

**No lifecycle scripts.** npm runs `preinstall`/`postinstall` scripts automatically. rusk doesn't. If a package depends on a postinstall step, you may need to run it manually or configure it in `[build]`.

**Digest pinning.** `rusk.lock` pins every package with its SHA-256 digest. This means `rusk install` on a different machine always gets the exact same bytes, even if the registry silently replaces an artifact.

**Virtual environment.** For Python projects, rusk materializes into `.venv/lib/site-packages/`. If you were installing globally with pip, you'll need to use the venv.
